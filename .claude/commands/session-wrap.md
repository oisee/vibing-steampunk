# Session Wrap - End-of-Session Feedback Extraction

You are performing an end-of-session wrap-up that extracts structured learnings from the current conversation. This generates a comprehensive session feedback log and routes action items to the right destinations.

## Step 1: Review Conversation Context

Analyze the full conversation to extract:

1. **Session objective** - What was the user trying to accomplish?
2. **Tools used** - Which VSP/MCP tools were called? (List tool names)
3. **Errors encountered** - Any tool failures, timeouts, or unexpected results
4. **Workarounds applied** - Creative solutions to problems hit during the session
5. **Gaps identified** - Missing tools, features, or capabilities that would have helped
6. **Bugs found** - Tool misbehavior, incorrect results, crashes
7. **Lessons learned** - Reusable techniques, best practices, or patterns discovered
8. **Opportunities** - Enhancement ideas, workflow improvements

## Step 2: Read Existing Session Feedback

Check `feedback/` directory for any feedback file from today's date (pattern: `YYYY-MM-DD-*-session-*.md`).

If found, read it to incorporate any `/feedback` entries already captured during the session. Avoid duplicating these in the auto-extracted entries.

## Step 3: Generate Session Feedback Log

Create or update the session feedback file at `feedback/YYYY-MM-DD-NNN-session-<topic>.md`.

If the file already exists (from `/feedback` calls), append the auto-extracted entries and the summary sections below. If creating new, use the full template.

### Full Template

```markdown
# Session Feedback Log

**Date:** YYYY-MM-DD
**Session ID:** NNN
**Objective:** [What was the user working on]
**Duration:** [Estimate based on conversation length]

---

## Entries

[All feedback entries - both manually captured via /feedback AND auto-extracted]

### Entry N [HH:MM]
- **Type:** [Bug | Gap | Opportunity | Lesson | Issue]
- **Severity:** [Critical | High | Medium | Low]
- **Component:** [component area]
- **Description:** [description]
- **Workaround:** [if applicable]
- **Action:** Pending

---

## Action Items

| # | Type | Description | Destination | Status |
|---|------|-------------|-------------|--------|
| 1 | [type] | [short description] | GitHub Issue / Backlog / CLAUDE.md | [ ] Pending |

## Session Metrics

- **Tools used:** [count unique tools]
- **Estimated tool calls:** [rough count from conversation]
- **Errors encountered:** [count]
- **Workarounds found:** [count]
- **Key tools:** [list the 3-5 most-used tools]

## Patterns & Observations

[2-3 sentences about what worked well, what didn't, and any recurring themes]
```

## Step 4: Update Backlog

Read `feedback/_BACKLOG.md` and append new action items from this session.

For each action item:
- Check if a similar item already exists in the backlog (same component + similar description)
- If duplicate: update the existing item with a note "Also seen in session YYYY-MM-DD-NNN"
- If new: add to the "Open Items" table

Use this format for new backlog entries:

```markdown
| YYYY-MM-DD | NNN | [Type] | [Description] | [Severity] | Open |
```

## Step 5: Route Action Items

Present the action items to the user and ask what to do with each batch:

### Bugs (severity >= High)
Ask: "Create GitHub issues for these N high-severity bugs?"
- If yes: for each bug, run:
  ```
  gh issue create --title "Bug: [title]" --body "$(cat <<'EOF'
  ## VSP Bug Report

  **Source:** Session feedback `feedback/[filename]`
  **Severity:** [severity]
  **Component:** [component]

  ### Description
  [description]

  ### Workaround
  [workaround]

  ---
  *Auto-generated by VSP feedback loop (`/session-wrap`)*
  EOF
  )" --label bug
  ```
- Update the action item status to: `Issue #[number]`

### Gaps and Opportunities
Ask: "Create GitHub issues for these N feature gaps/opportunities?"
- If yes: create issues with label `enhancement`
- If no: confirm they'll stay in `_BACKLOG.md`

### Lessons
Ask: "Update CLAUDE.md with these N lessons?"
- If yes: identify the right section(s) in CLAUDE.md and append the lessons
- Common target sections: "Common Issues", "Code Patterns", "Testing", or a new subsection if appropriate
- Update action item status to: `Updated CLAUDE.md`

## Step 6: Commit Feedback Log

Ask the user: "Commit this session's feedback log?"

If yes, stage and commit:
```bash
git add feedback/YYYY-MM-DD-NNN-session-*.md feedback/_BACKLOG.md
git commit -m "feedback: session YYYY-MM-DD-NNN - [brief objective summary]"
```

## Step 7: Print Summary

Display a concise summary:

```
Session Wrap Complete
─────────────────────
Objective: [objective]
Duration:  [estimate]
Entries:   N total (X bugs, Y gaps, Z lessons, W opportunities)
Actions:   N routed (X issues created, Y backlog items, Z CLAUDE.md updates)
Feedback:  feedback/[filename]
Backlog:   feedback/_BACKLOG.md ([total open items] open items)
```

## Important Notes

- Be thorough but concise in descriptions - these should be actionable by someone who wasn't in the session
- For auto-extracted entries, set the time to "auto" instead of a specific timestamp
- If no significant feedback was generated during the session, say so - don't force entries
- Severity assessment should consider impact on VSP's value proposition (tools that fail during critical workflows are High+)
- The session topic for the filename should reflect the PRIMARY activity, not every tangent
