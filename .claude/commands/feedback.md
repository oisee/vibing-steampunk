# Quick Feedback Capture

You are capturing structured feedback about VSP during an active session. This is designed to be fast (under 30 seconds) so it doesn't break flow.

## Step 1: Gather Feedback

Ask the user these questions using AskUserQuestion (all in a single call):

1. **Type** - What kind of feedback?
   - Bug (tool misbehavior, errors, crashes)
   - Gap (missing tool, missing feature, capability limitation)
   - Opportunity (enhancement idea, workflow improvement)
   - Lesson (reusable technique, workaround, best practice discovered)
   - Issue (documentation gap, confusing behavior, UX problem)

2. **Severity** - How impactful?
   - Critical (blocks workflow, data loss risk)
   - High (significant friction, requires workaround)
   - Medium (annoying but manageable)
   - Low (nice-to-have improvement)

3. **Component** - Which area?
   - ATC (code checks)
   - Search (object search, grep)
   - CodeIntel (call graph, structure, find refs)
   - CRUD (create, update, delete, activate)
   - Transport (CTS, transport management)
   - OData/RAP (service binding, CDS)
   - Debugger (breakpoints, stepping, variables)
   - CLI (command line, config, profiles)
   - Git (abapGit integration)
   - Reports (report execution, variants)
   - UI5 (Fiori/BSP management)
   - Safety (read-only, package restrictions)
   - Other

## Step 2: Get Description

After the structured questions, ask the user for a free-text description. Prompt them:

> Describe the feedback briefly. Include: what happened, what you expected, and any workaround you found (if applicable).

## Step 3: Write Feedback Entry

Determine the current session feedback file:
- Check `feedback/` directory for any file matching today's date pattern: `YYYY-MM-DD-NNN-session-*.md`
- If no file exists for today, create one. To determine NNN: count existing files for today's date and increment. For the topic suffix, use a short kebab-case summary of the current conversation's main activity (e.g., `session-atc-analysis`, `session-debugger-testing`, `session-migration-review`).

### If creating a new session file

Write the header:

```markdown
# Session Feedback Log

**Date:** YYYY-MM-DD
**Session ID:** NNN
**Objective:** [Infer from conversation context - what is the user working on?]

---

## Entries

```

### Append the feedback entry

Append to the session file:

```markdown
### Entry N [HH:MM]
- **Type:** [selected type]
- **Severity:** [selected severity]
- **Component:** [selected component]
- **Description:** [user's description]
- **Workaround:** [if provided, otherwise "None identified"]
- **Action:** Pending

---
```

## Step 4: Route Based on Type and Severity

Apply these routing rules:

### Bugs with severity >= High
Ask the user: "This is a high-severity bug. Create a GitHub issue now?"
- If yes: run `gh issue create` with this template:
  ```
  gh issue create --title "Bug: [short title]" --body "$(cat <<'EOF'
  ## VSP Bug Report

  **Source:** Session feedback `feedback/[filename]`
  **Severity:** [severity]
  **Component:** [component]

  ### Description
  [description]

  ### Expected Behavior
  [from description]

  ### Workaround
  [workaround if any]

  ---
  *Auto-generated by VSP feedback loop (`/feedback`)*
  EOF
  )" --label bug
  ```
- Update the entry's Action field to: `GitHub Issue #[number]`

### Lessons (any severity)
Ask the user: "This is a useful lesson. Update CLAUDE.md with this learning?"
- If yes: identify the appropriate section in CLAUDE.md (usually "Common Issues" or a relevant section) and append the lesson.
- Update the entry's Action field to: `Updated CLAUDE.md`

### Gaps and Opportunities
Tell the user: "Logged to session feedback. Will be added to `_BACKLOG.md` during `/session-wrap`."

## Step 5: Confirm

Count total entries in the current session file and report:

> **Logged.** Entry #N recorded in `feedback/[filename]`.
> Session total: N entries (X bugs, Y gaps, Z lessons).
> [If issue created: "GitHub Issue #NNN created."]
> [If CLAUDE.md updated: "CLAUDE.md updated with new lesson."]
